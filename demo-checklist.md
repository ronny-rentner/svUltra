# Demo Checklist

Before presenting or building demo/UI changes, every item below must be green.

- [ ] Visible demo, prose, and code examples demonstrate the same feature.
- [ ] Every label names something real on screen.
- [ ] Every code comparison matches the rendered example structurally.
- [ ] Every custom style is necessary, nested, and uses PicoCSS variables where styling is needed.
- [ ] Every wrapper, class, constant, and component has a concrete reason tied to the demo goal.
- [ ] Every visible/code construct is part of the feature story, not demo scaffolding exposed as the lesson.
- [ ] Every demonstrated style creates a visible, meaningful difference from the default.
- [ ] Every visual change is obvious at normal viewing size.
- [ ] Every component style demonstrates parent-level theming, not an arbitrary one-off restyle that should be a component variant.
- [ ] Every framework variable or API used in examples is real and already established by the project or verified from the framework source; do not invent variable names.
- [ ] Re-read the full page as a product demo, not just as compiling Svelte.
